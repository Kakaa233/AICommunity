这是一份针对 Wego 项目注册登录模块的重构设计文档。文档从架构工程师的视角，详细规划了从“手机号/验证码”向“用户名/密码”体系迁移的设计方案。

# Wego 项目注册/登录模块重构架构文档

## 1. 重构背景与目标
*   **当前痛点**：由于原有系统过度依赖手机号与短信验证码机制，导致存在短信任务成本、外部依赖稳定性风险，且限制了未绑定手机号用户的访问。
*   **重构目标**：彻底剥离手机号获取、短信发送及验证码强校验逻辑。转为标准的“用户名 + 密码”认证体系，精简冗余代码（如第三方短信 SDK 及 OAuth 授权模块），提升系统内聚性与注册转化率。

## 2. 核心规则设计
为了保证系统的规范性与安全性，对新的注册登录载体（用户名与密码）制定以下规则：

### 2.1 账户名（Username / UserId）规则
*   **唯一性**：全局唯一，作为识别用户的核心主键（复用现有数据库中的 `user_id` 字段）。
*   **长度限制**：5 ~ 20 个字符。
*   **字符限制**：仅允许包含大小写字母、数字以及下划线（`^[a-zA-Z0-9_]{5,20}$`）。不支持纯数字，以防与历史遗留的真实手机号冲突。
*   **不可变性**：一旦注册成功，禁止修改。

### 2.2 密码（Password）规则
*   **长度限制**：8 ~ 20 个字符。
*   **复杂度**：必须包含至少一个字母和一个数字（`^(?=.*[A-Za-z])(?=.*\d)[A-Za-z\d]{8,20}$`）。
*   **持久化验证**：后端必须配合随机 Salt（盐值）进行多次 MD5 散列后落库，禁止明文存储。

## 3. 前端重构方案 (Vue + Element UI)

前端需要从 UI 交互和路由层面进行裁剪与改造，主要涉及 src 目录下的视图配置。

### 3.1 路由层调整
*   **移除多余路由**：删除基于验证码登录的路由入口。在 index.js 中，彻底移除对 `loginPass.vue` 文件的引用及路由规则。
*   **保留核心路由**：仅持有 `/login` (密码登录) 和 `/register` (用户注册)。

### 3.2 视图层改造 (Views)
*   **register.vue**:
    *   **字段替换**：表单中的 `<el-input>` 收权，将 `phone` (手机号) 字段替换为 `userId` (账号名)。
    *   **移除交互**：移除“获取验证码”按钮及其关联的倒计时计时器方法、移除“输入验证码”输入框。
    *   **防刷机制（可选）**：为弥补失去短信验证带来的防机器人注册风控下降，建议在注册按钮前引入**图形验证码 (Captcha)** 或滑动滑块验证组件。
*   **login.vue**:
    *   **字段替换**：将登录凭证字段由 `phone` 修改为 `userId`。
    *   **UI 简化**：精简登录界面的第三方登录入口（去除 QQ/Github 图标及回调链路）。

---

## 4. 后端重构方案 (Spring Boot)

后端重点在于移除外部短信服务的集成，并更改校验拦截器的行为。

### 4.1 控制器改造 (Controllers)
*   **RegisterController.java**:
    *   清理原有的 `sendSms` 分发接口。
    *   核心防重逻辑：在 `registerVerify` 中，使用 `userService.selectByUserId(userId)` 检查数据库以实施全局唯一性校验。
    *   默认属性填充：由于缺少了其他维度的信息（如手机归属地等），注册时由后端自动生成默认的随机昵称（如 "用户" + 随机字符串）、默认头像等，提高注册体验。
*   **LoginController.java**:
    *   只保留接收 `@RequestParam String userId` 和 `@RequestParam String password` 的标准 POST 接口。
    *   数据库比对成功后，签发并返回 JWT 或 Session Token。

### 4.2 实体与服务层层简化
*   **废弃短信 Service**：移除处理腾讯云/阿里云短信下发的组件。
*   **清理 Pom 依赖**：可在 pom.xml 中安全剔除诸如 `tencentcloud-sdk-java` 等不再使用的第三方发信依赖，减小打包体积。

---

## 5. 数据库与数据迁移考量 (MySQL)

为避免对核心表 `User` 做破坏性重建，我们将采取**平滑过渡**的策略：

1.  **复用字段**：不再新增 `username` 字段，直接将原具有唯一索引属性的 `user_id` 作为账号名进行存取。
2.  **约束放宽**：取消 `phone` 字段的 `NOT NULL` (非空) 约束。在重构后的注册流中，新用户的 `phone` 字段将默认插入 `NULL` 或空字符串。
3.  **历史数据兼容**：老用户依然可以使用曾经绑定的手机号作为 `userId` 名进行登录。新规则中“禁止纯数字注册”的限制恰好可以确保新用户的自定义账号名绝对不会与老用户的手机号发生冲突。

---

## 6. 后续演进建议
此重构版本将系统的准入门槛降至最低，随后应考虑引入**系统内双态绑定策略**：
在用户登录成功后的个人中心 (my_info.vue) 提供“绑定邮箱”功能。对于忘记密码的场景，不依赖运营人员介入，可通过已绑定的邮箱收取安全链接完成自主找回密码，从而达到最终闭环。
